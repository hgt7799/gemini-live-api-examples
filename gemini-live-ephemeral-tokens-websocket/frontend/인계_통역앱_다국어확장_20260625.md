# 작업 인계: 실시간 통역앱 다국어 확장 (메인 페이지 + 언어 선택)

## 0. 대화 방식 (필수)
- 존댓말로 대화.
- 작업 종료/대화 끊기 제안 절대 금지. 종료는 내가 명시할 때만.
- 터미널 작업: 한 번에 한 명령어씩. bash 코드블록 + 예상 결과 별도 코드블록, "칠 명령어" vs "예상 결과" 라벨 구분.
- 작업 한 세트 = [코드 수정 -> tsc/문법확인 -> diff -> 파일지정 add -> 커밋 -> push -> 인계 md 저장]. 코드 커밋만으로 끝내지 말 것.
- "여긴 컨테이너라 레포 접근 못 한다" 절대 금지. CC에 명령 -> 내가 출력 붙여주는 방식이 표준.
- 작업 대원칙: 무조건 정확/자세/조심/신중. "고쳤다"가 아니라 "고친 게 맞다"까지 테스트로 확인.
- CC가 Read 도구로 요약만 줄 때가 있음. 정확한 원문 필요시 cat -n 요청 (한글 깨지니 -ev/-A 쓰지 말 것).

## 3. 확인된 모델 제약 (재검토 금지 - 이미 공식 문서로 확인)
- 모델: gemini-3.5-live-translate-preview (전용 번역 모델, Gemini 3 Pro 기반).
- 이 모델은 시스템 프롬프트를 안 받음. translation_config 옵션은 targetLanguageCode / echoTargetLanguage 둘뿐.
- 짧은 말("오늘 점심")을 자연스러운 문장으로 늘리거나 안 들은 내용 채우는 현상 -> 수정 불가 (연속 스트림 처리 = 모델 설계). 직역 원하면 turn 기반 모델 교체해야 하나 대공사라 보류.
- 입력 오디오 16kHz PCM mono, 출력 24kHz PCM mono.

## 4. 이번 작업 목표
지금 도착 언어가 영어 고정 (server.py TARGET_LANGUAGE_CODE = "en"). 이걸 풀어서:
1. 메인 페이지(언어 선택 화면) 신규 제작 - 앱 켜면 먼저 "어떤 언어로 통역?" 선택 -> 통역 화면 이동.
2. server.py - 토큰에 언어를 박지 말고 unlock 발급. (공식 문서: 토큰 생성 시 translationConfig를 빼고 "lock_additional_fields": [] 로 설정 -> 클라이언트에서 도착 언어 설정 가능.)
3. geminilive.js - 선택된 도착 언어를 setup 메시지(translationConfig.targetLanguageCode)에 실어 전송.
4. script.js - 메인에서 선택한 언어 받아서 연결.
-> 이번엔 통역 로직(server.py, geminilive.js, script.js) 수정 허용(다국어 확장 목적). mediaUtils.js는 안 건드림.

## 5. 지원 언어 (메인 페이지 버튼 재료)
- 모델은 70개 이상 언어 자동 감지. 도착 언어로 BCP-47 코드 지정(예: en, ja, zh-Hans, es, fr, de, it, pt-BR, hi, ar, ko 등).
- 주의: 70개 전체 언어 코드 공식 표는 아직 미확보. 버튼 깔기 전 실제 동작하는 언어 코드 확인 필요(추측으로 깔지 말 것). 검증된 주요 언어부터 시작, 점진 확장 권장.

## 6. 시작 지점
먼저 server.py 현재 토큰 발급 로직(24~75행 부근, translation_config 박는 부분)을 cat -n으로 정확히 보고, unlock 방식 설계부터. 추측으로 코드 안 건드리고 현재 코드 확인 후 진행.
새 창에서는 index.html(현재 디자인)을 따로 업로드해서 볼 것.
